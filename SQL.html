Complete Interview Cheat Sheet - Key Differences Explained

1. Primary Key vs Unique Key

Primary Key vs Unique Key
Primary Key Unique Key
Uniquely identifies each row Ensures column values are unique
Only one primary key allowed per table Multiple unique keys allowed per table
Does not allow NULL values Allows one NULL value in a column
Creates a clustered index by default Creates a non-clustered index by default
Used to enforce entity integrity Used to enforce uniqueness without entity constraint
-- Example
CREATE TABLE Students (
  ID INT PRIMARY KEY,
  Email VARCHAR(100) UNIQUE
);

2. Clustered Index vs Non-Clustered Index

Clustered vs Non-Clustered Index
Clustered Index Non-Clustered Index
Sorts and stores the actual data rows physically Stores a separate structure with pointers to data rows
Only one clustered index allowed per table Multiple non-clustered indexes allowed
Faster for range queries (BETWEEN, >, <) Faster for exact match queries (= operator)
Changes the physical order of data on disk Does not change physical data order
-- Clustered index usually created with PRIMARY KEY
CREATE TABLE Employees (
  EmployeeID INT PRIMARY KEY, -- Clustered Index
  Name VARCHAR(100)
);

-- Creating non-clustered index
CREATE NONCLUSTERED INDEX idx_Name ON Employees(Name);

3. DELETE vs TRUNCATE vs DROP

DELETE vs TRUNCATE vs DROP
DELETE TRUNCATE DROP
Deletes specific rows (can use WHERE clause) Deletes all rows from the table quickly Deletes the entire table structure and data
Logs deletion of each row individually Logs deallocation of data pages (less logging) Removes table metadata and data permanently
Can be rolled back within a transaction Usually cannot be rolled back Cannot be rolled back once executed
Triggers are fired on DELETE Triggers do not fire Triggers removed with table
DELETE FROM Employees WHERE Department = 'Sales';
TRUNCATE TABLE Employees;
DROP TABLE Employees;

4. WHERE vs HAVING

WHERE vs HAVING
WHERE HAVING
Filters individual rows before grouping Filters groups after aggregation
Cannot use aggregate functions Can use aggregate functions to filter groups
Used with SELECT, UPDATE, DELETE Used mainly with GROUP BY in SELECT
Generally faster (filters early) Filters results after groups are formed
-- Using WHERE to filter rows before grouping
SELECT Department, COUNT(*)
FROM Employees
WHERE Salary > 50000
GROUP BY Department;

-- Using HAVING to filter groups after aggregation
SELECT Department, COUNT(*)
FROM Employees
GROUP BY Department
HAVING COUNT(*) > 5;

5. INNER JOIN vs LEFT JOIN vs RIGHT JOIN vs FULL JOIN

Types of JOINs
INNER JOIN LEFT JOIN RIGHT JOIN FULL JOIN
Returns only rows with matching values in both tables Returns all rows from left table + matching rows from right Returns all rows from right table + matching rows from left Returns all rows from both tables
Non-matching rows are excluded Non-matching right table rows return NULL Non-matching left table rows return NULL Non-matching rows from both sides return NULL
Usually used for strict matching Want all left side data regardless of matches Want all right side data regardless of matches Want complete data from both tables
SELECT * FROM A INNER JOIN B ON A.ID = B.ID;
SELECT * FROM A LEFT JOIN B ON A.ID = B.ID;
SELECT * FROM A RIGHT JOIN B ON A.ID = B.ID;
SELECT * FROM A FULL JOIN B ON A.ID = B.ID;

6. UNION vs UNION ALL

UNION vs UNION ALL
UNION UNION ALL
Removes duplicate rows from the combined result set Includes all rows, including duplicates
Slower performance due to duplicate elimination Faster performance; no duplicate checking
Requires sorting or hashing internally Simply appends result sets without sorting
Useful when you want distinct results only Useful when duplicates are acceptable
SELECT City FROM Customers
UNION
SELECT City FROM Suppliers;

SELECT City FROM Customers
UNION ALL
SELECT City FROM Suppliers;

7. Subquery vs Join

Subquery vs Join
Subquery Join
A query nested inside another query Combines rows from two or more tables
Can return single value or table Returns combined rows as a result set
Sometimes less efficient Generally more efficient for large datasets
Easier to write for simple conditions Better for complex queries involving multiple tables
-- Subquery
SELECT Name FROM Employees
WHERE DepartmentID = (SELECT ID FROM Departments WHERE Name = 'HR');

-- Join
SELECT Employees.Name, Departments.Name
FROM Employees
JOIN Departments ON Employees.DepartmentID = Departments.ID;

8. CHAR vs VARCHAR

CHAR vs VARCHAR
CHAR VARCHAR
Fixed length data type Variable length data type
Pads extra spaces to fill the length Stores only the entered characters
Uses fixed amount of storage Uses storage based on actual data size
Faster for fixed-size data More flexible and space-efficient
-- CHAR(10) stores exactly 10 characters
DECLARE @charExample CHAR(10) = 'Hello';

-- VARCHAR(10) stores only characters entered
DECLARE @varcharExample VARCHAR(10) = 'Hello';

9. Stored Procedure vs Function

Stored Procedure vs Function
Stored Procedure Function
Can perform DML and DDL Limited DML allowed
Does not have to return a value Must return a value
Can call other procedures and functions Can call only other functions
Supports transaction control Limited transaction control
-- Stored Procedure
CREATE PROCEDURE usp_GetEmployeeCount
AS
BEGIN
  SELECT COUNT(*) FROM Employees;
END;

-- Function
CREATE FUNCTION dbo.fn_GetEmployeeCount()
RETURNS INT
AS
BEGIN
  RETURN (SELECT COUNT(*) FROM Employees);
END;

10. View vs Table

View vs Table
View Table
Virtual table based on a SELECT query Physical storage of data
Does not store data physically Stores data physically in database
Used for abstraction and security Used for storing actual data
Data is dynamic Data is static until updated
-- Create a view
CREATE VIEW vw_ActiveEmployees AS
SELECT * FROM Employees WHERE Status = 'Active';

-- Create a table
CREATE TABLE Employees (
  EmployeeID INT PRIMARY KEY,
  Name VARCHAR(100),
  Status VARCHAR(10)
);

11. Temporary Table vs Table Variable

Temp Table vs Table Variable
Temporary Table Table Variable
Created and stored in tempdb database Stored in memory (more lightweight)
Supports indexes, constraints, and statistics Limited support for indexes and constraints
Can be used across multiple batches Scope limited to the batch or procedure
Can handle large amounts of data Best suited for small datasets
-- Temporary Table
CREATE TABLE #TempEmployees (ID INT, Name VARCHAR(50));

-- Table Variable
DECLARE @TempEmployees TABLE (ID INT, Name VARCHAR(50));

12. Cross Join vs Inner Join

Cross Join vs Inner Join
Cross Join Inner Join
Returns Cartesian product Returns only rows with matching values
No ON or JOIN condition needed Requires ON condition
Result size = rows A × rows B Result size depends on matching rows
Used when all combinations needed Used for filtering related data
-- Cross Join (Cartesian product)
SELECT * FROM Employees CROSS JOIN Departments;

-- Inner Join
SELECT * FROM Employees INNER JOIN Departments ON Employees.DeptID = Departments.ID;

13. Primary Key vs Foreign Key

PK vs FK
Primary Key Foreign Key
Uniquely identifies each row in a table Creates a link between two tables
Does not allow NULL Can allow NULL values
Only one per table Multiple per table
Enforces entity integrity Enforces referential integrity
CREATE TABLE Departments (
  DeptID INT PRIMARY KEY,
  DeptName VARCHAR(100)
);

CREATE TABLE Employees (
  EmployeeID INT PRIMARY KEY,
  Name VARCHAR(100),
  DeptID INT,
  FOREIGN KEY (DeptID) REFERENCES Departments(DeptID)
);

14. DDL vs DML vs DCL vs TCL

SQL Command Types
DDL DML DCL TCL
CREATE, ALTER, DROP SELECT, INSERT, UPDATE, DELETE GRANT, REVOKE COMMIT, ROLLBACK, SAVEPOINT
Defines database structure Manipulates data Manages permissions Manages transactions
Affects metadata Affects actual data Security control Atomic operations
Auto-committed Can be rolled back Permission commands Transaction control

15. Rank() vs Dense_Rank() vs Row_Number()

Window Functions
Rank() Dense_Rank() Row_Number()
Gaps for ties (1,2,2,4) No gaps (1,2,2,3) Unique numbers (1,2,3,4)
Skips numbers after ties Continues without skipping Always increments by 1
Useful for ranking with gaps Useful without gaps Useful for unique numbering
SELECT EmployeeID, Salary,
      RANK() OVER (ORDER BY Salary DESC) AS Rank,
      DENSE_RANK() OVER (ORDER BY Salary DESC) AS DenseRank,
      ROW_NUMBER() OVER (ORDER BY Salary DESC) AS RowNum
FROM Employees;

16. OLTP vs OLAP

OLTP vs OLAP
OLTP OLAP
Day-to-day transaction processing Complex analytical queries
Short, fast transactions Fewer, long-running queries
Highly normalized Often denormalized
Real-time data Historical data analysis

17. Normalization vs Denormalization

Normalization vs Denormalization
Normalization Denormalization
Reduce redundancy Improve read performance
Multiple related tables Fewer tables
More joins Less joins
Maintain consistency Can cause anomalies

18. CTE vs Subquery

CTE vs Subquery
CTE Subquery
Temporary named result set Query nested inside another
Improves readability Can be harder to read
Can be recursive Cannot be recursive
Reusable in query Typically used once
-- CTE example
WITH EmployeeCTE AS (
  SELECT EmployeeID, ManagerID FROM Employees
)
SELECT * FROM EmployeeCTE;

-- Subquery example
SELECT EmployeeID FROM Employees
WHERE ManagerID = (SELECT ManagerID FROM Managers WHERE Name = 'John');

19. Index Scan vs Index Seek

Index Scan vs Seek
Index Scan Index Seek
Reads all rows in index Navigates to specific rows
Slower for large tables Faster, efficient
High I/O and CPU Low I/O and CPU

20. Schema vs Database

Schema vs Database
Schema Database
Logical container for objects Physical container for data
Organizes and secures objects Manages storage and security
Multiple per database Standalone entity
Avoids naming conflicts Contains multiple schemas
